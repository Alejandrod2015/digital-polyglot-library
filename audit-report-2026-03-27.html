CRITICAL — Rompe funcionalidad o expone datos

API-001 Audio generation sin autenticación
src/app/api/audio/generate/route.ts
El endpoint POST genera audio vía ElevenLabs TTS sin verificar auth. Cualquier persona puede disparar generación de audio costosa sin estar logueada.
Todos — endpoint público
Agregar const { userId } = await auth(); if (!userId) return 401; al inicio del handler.
API-001 Synopsis generation sin autenticación
src/app/api/generate-synopsis/route.ts
POST llama a OpenAI GPT-4o-mini para generar sinopsis sin auth. Riesgo de abuso y costos no controlados.
Todos — endpoint público
Agregar auth check antes de la llamada a OpenAI.
API-001 Traducción sin autenticación
src/app/api/translate/route.ts
POST llama a DeepL API sin auth. Cuota de traducción puede ser agotada por cualquiera.
Todos — endpoint público
Agregar auth gate.
API-002 Studio create-stories expone datos de todos los usuarios
src/app/api/studio/create-stories/route.ts
GET retorna todas las user stories (userId, títulos, idioma, nivel, tópico) sin autenticación. Expone metadata de contenido personal de todos los usuarios.
Todos los usuarios con stories creadas
Agregar auth y filtrar por userId en la query.
API-005 Shopify webhook con HMAC deshabilitado
src/app/api/shopify/webhook/route.ts
La validación HMAC está comentada con "ignorando para test". En producción, cualquiera puede crear claim tokens para emails y libros arbitrarios sin compra válida de Shopify.
Sistema de billing completo
Re-habilitar validación HMAC. Nunca bypassear en producción.
USR-001 Usuarios anónimos ven links a páginas protegidas
src/components/Sidebar.tsx · src/components/MobileMenu.tsx · src/components/MobileTabBar.tsx
Usuarios no logueados ven links de navegación a /my-library, /favorites, /progress, y /journey. Al visitarlas, la experiencia es inconsistente: favorites muestra "Sign in", progress muestra "Sign in", pero journey intenta cargar datos que requieren sesión.
Usuarios anónimos
Envolver estos links en <SignedIn> en Sidebar, MobileMenu y MobileTabBar. O agregar redirects server-side en las páginas.
USR-001 Journey page sin auth guard pero depende de datos de usuario
src/app/journey/page.tsx (línea ~28)
Llama a currentUser() sin verificar null. Usa user?.publicMetadata y pasa a funciones que esperan sesión activa (getCompletedJourneyStoryKeys, getJourneyDueReviewItems). Usuario anónimo obtiene comportamiento indefinido.
Usuarios anónimos
Agregar if (!user) redirect("/sign-in") después de currentUser().
BILL-001 Conflicto de suscripción dual sin resolución
prisma/schema.prisma · src/app/api/stripe/webhook/route.ts · src/app/api/billing/google-play/verify/route.ts
BillingEntitlement tiene constraint unique en userId (una fila por usuario). Si un usuario tiene suscripción Stripe Y Google Play, el último webhook que llega sobreescribe al otro. El usuario podría estar pagando doble mientras solo una suscripción está activa en la DB.
Usuarios con suscripciones en múltiples plataformas
Cambiar constraint a @@unique([userId, source]), verificar si existe otra suscripción antes de upsert, cancelar la vieja si se cambia de fuente.
BILL-002 Clerk metadata puede desincronizarse de la DB
src/lib/billingClerk.ts · src/app/api/stripe/webhook/route.ts
Después de actualizar BillingEntitlement en la DB, se llama syncClerkPlanFromEntitlement() para sincronizar Clerk. Si la sincronización falla (error de red, rate limit), la DB está actualizada pero Clerk queda stale. No hay retry ni mecanismo de recuperación. El usuario ve el plan viejo hasta re-login.
Cualquier usuario que cambie de plan
Agregar retry con exponential backoff, timestamp de última sincronización, y job periódico que verifique que Clerk coincide con la DB.
ERR-001 Libros/historias inexistentes no retornan 404
src/app/books/[bookSlug]/page.tsx (línea ~20) · src/app/books/[bookSlug]/[storySlug]/page.tsx (línea ~136)
Cuando un libro o historia no existe, retorna un <div> con "Libro no encontrado" en vez de llamar notFound(). No hay 404 HTTP, no hay fallback de Next.js, y el layout completo (sidebar, nav) sigue renderizando.
Cualquiera con URL inválida
Reemplazar los returns de div con notFound().
ERR-003 Múltiples fetches client-side sin manejo de errores
src/app/books/[bookSlug]/ReaderClient.tsx · src/app/practice/page.tsx · src/app/create/page.tsx · src/app/HomeClient.tsx
Llamadas fetch() sin try-catch ni estado de error. Fallos de red fallan silenciosamente — el usuario no recibe feedback.
Todos los usuarios con conexión inestable
Envolver todos los fetch en try-catch con estado de error visible al usuario.

WARNING — UX confusa o inconsistente

USR-003 Story of Week/Day: nav filtrada por plan pero páginas son públicas
src/components/Sidebar.tsx (líneas 215-232) · src/app/story-of-the-week/page.tsx · src/app/story-of-the-day/page.tsx
Sidebar muestra "Story of the Week" solo a plan free, y "Story of the Day" solo a plan basic. Pero ambas páginas son accesibles por URL para cualquiera sin restricción de plan.
Todos los tipos de usuario
O mostrar ambos links a todos, o agregar guards server-side en las páginas.
USR-004 Settings accesible para anónimos pero cambios no persisten
src/components/Sidebar.tsx (líneas 245-251) · src/app/settings/page.tsx
Link de Settings visible para todos. Usuarios anónimos pueden cambiar tema localmente, pero cambios de idioma/región/plan no se guardan server-side. UX engañosa.
Usuarios anónimos
Esconder Settings de anónimos o mostrar "Sign in para guardar cambios".
API-004 Claim tokens sin validación de expiración
src/app/api/claim/[token]/route.ts · prisma/schema.prisma
El schema tiene campo expiresAt pero el endpoint nunca lo verifica. El Shopify webhook tampoco lo setea al crear tokens. Todos los tokens son válidos indefinidamente.
Sistema de claims
Verificar expiresAt en el endpoint y setear expiración (ej: 90 días) al crear tokens.
BILL-003 Trial expirado no se actualiza en DB
src/lib/billing.ts · src/app/api/billing/entitlement/route.ts
Expiración de trial se calcula on-the-fly (trialEndsAt > now()). El status en DB queda "trialing" para siempre. Si un webhook se pierde, el estado nunca transiciona a "expired".
Usuarios en trial
Agregar cron job que actualice status de "trialing" a "expired" cuando trialEndsAt < NOW().
BILL-004 Race condition en redención de claim tokens
src/app/api/claim/[token]/route.ts (líneas 75-96)
Entre el check de claim.redeemedBy y el update, dos requests concurrentes pueden redimir el mismo token. No hay transacción ni lock de base de datos.
Usuarios redimiendo tokens simultáneamente
Usar transacción Prisma con isolation level SERIALIZABLE, o conditional update atómico.
ERR-005 Estados vacíos inconsistentes en páginas de listas
src/app/my-library/MyLibraryClient.tsx · src/app/explore/ExploreClient.tsx · src/app/favorites/page.tsx
Cuando no hay datos (librería vacía, sin favoritos, búsqueda sin resultados), no siempre se muestra mensaje significativo. Si el API falla y datos offline están vacíos, el usuario ve página en blanco.
Usuarios nuevos, usuarios con errores de red
Agregar UI de estado vacío explícito con mensaje y CTA claro.
ERR-004 Páginas con data fetching sin loading.tsx
src/app/my-library/ · src/app/favorites/ · src/app/progress/ · src/app/settings/ · src/app/plans/ · src/app/create/
6 páginas fetchean datos en cliente pero no tienen loading.tsx ni skeleton UI. Flash de pantalla en blanco durante carga inicial.
Todos los usuarios (especialmente conexiones lentas)
Crear loading.tsx con skeleton loaders para cada una de estas páginas.
NAV-001 Back navigation rota para rutas no trackeadas
src/components/BackButton.tsx · src/components/BackNavigationHandler.tsx
BackNavigationHandler solo trackea secciones principales (/explore, /my-library, /favorites, /settings). Navegando desde /explore → /books/xxx/chapter-1 → botón back va a / en vez de /explore.
Todos los usuarios navegando libros/stories
Agregar tracking para /books/, /stories/, /journey/ en BackNavigationHandler.
PERF-001 Imágenes sin next/image optimization
src/app/explore/ExploreClient.tsx · src/app/HomeClient.tsx · src/components/StoryVerticalCard.tsx · src/components/BookStoriesGrid.tsx · src/components/Cover.tsx
Tags <img> crudos en vez de next/image. Sin lazy loading automático, sin conversión a AVIF/WebP, sin responsive sizing.
Performance para todos los usuarios
Reemplazar <img> con <Image> de next/image con width, height, y fill props.
ERR-001 Studio routes sin validación de existencia de recursos
src/app/studio/journey-builder/[language]/[variantId]/page.tsx · src/app/studio/journey-stories/[storyId]/page.tsx
Validan permisos de usuario pero no validan que language/variant/storyId existan. IDs inválidos cargan UI vacía sin indicación de error.
Usuarios de studio con URLs inválidas
Validar existencia de recursos y llamar notFound() si no existen.
BILL-007 Lógica de sincronización Clerk inconsistente
src/lib/billingClerk.ts
Tres funciones diferentes para sincronizar Clerk (syncClerkPlanFromEntitlement, syncClerkStripeSubscription, syncClerkStripeCancellation) con diferentes estrategias de metadata update y sin error handling.
Usuarios cambiando de plan
Consolidar en una sola función syncClerkPlanFromDatabase(userId) que siempre lea fresh BillingEntitlement de la DB.
CNT-002 Debug console.logs con emojis en producción
src/app/api/shopify/webhook/route.ts
Múltiples console.log() con prefijos emoji ("🔍 RAW BODY", "🧩 LINE ITEMS", "🔑 HMAC recibido", etc.) que son claramente debugging statements, no logging legítimo.
Logs de producción
Reemplazar con logging condicional detrás de flag DEBUG_WEBHOOKS, o usar librería de logging.

CNT-003 Origins de desarrollo hardcodeados en CORS
src/lib/sanityCors.ts
localhost:3000, localhost:3333, 127.0.0.1 hardcodeados en el array de allowed origins para Sanity CORS.
Mover a variable de entorno ALLOWED_DEV_ORIGINS que solo se cargue fuera de producción.
PERF-002 63 directivas "use client" necesitan revisión
Múltiples archivos
63 componentes marcados como client. Algunos podrían ser server components, reduciendo bundle size.
Auditar cada "use client" para verificar que usa hooks, event handlers, o browser APIs.
BILL-005 Google Play token sin cache
src/lib/googlePlay.ts
getGooglePlayAccessToken() pide token OAuth nuevo en cada llamada. El token dura 1 hora pero nunca se cachea.
Cachear token con TTL de 55 minutos (exp - 5 min de margen).
BILL-006 Sin audit log para cambios de billing
src/app/api/stripe/webhook/route.ts · src/app/api/billing/google-play/rtdn/route.ts
Cambios de suscripción solo se loguean con console.log(). No hay trail persistente en la DB. Riesgo de compliance para transacciones financieras.
Crear tabla BillingAuditLog con userId, action, source, oldStatus, newStatus, timestamp.
USR-006 Sign-in inconsistente entre componentes mobile
src/components/MobileTabBar.tsx · src/components/MobileMenu.tsx
MobileTabBar reemplaza Settings tab con "Sign in" para anónimos. MobileMenu muestra "Sign in" en el header pero no reemplaza tabs. Comportamiento diferente entre componentes móviles.
Unificar la experiencia de sign-in en ambos componentes mobile.
